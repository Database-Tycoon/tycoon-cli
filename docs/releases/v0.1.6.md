# Tycoon v0.1.6

_Released: TBD_

## Headline

In-flight. The cycle's planned headline is **OSI semantic-layer scaffolding** — a `tycoon semantics scaffold` command that emits an [Open Semantic Interchange v0.1.1](https://open-semantic-interchange.org/) YAML from your dbt marts. dbt MetricFlow stays the source of truth; OSI is a portable, vendor-neutral *view* onto dbt that downstream tools (Snowflake, ThoughtSpot, BlackRock, RelationalAI) can consume directly.

The OSI work is preceded by a small dependency-hygiene cleanup carried over from the v0.1.5 audit (this is the first PR of the cycle).

Design discussion in progress in [`docs/proposals/v0.1.6-and-v0.1.7-open-questions.md`](../proposals/v0.1.6-and-v0.1.7-open-questions.md). Several open questions still need resolving before OSI code starts (command shape — new `semantics` namespace vs. `--osi` flag on `data analyze`; YAML file layout — per-mart vs. project-level; aggressiveness of the auto-aggregation heuristic).

## Themes (working list)

### Dependency hygiene (first PR of the cycle)

Drop the two redundant pins flagged in v0.1.5's `docs/reference/dependencies.md`:

- **`websockets==16.0`** from `[server]` — pulled transitively by `uvicorn[standard]==0.46.0`; tycoon never `import websockets` (uses fastapi's `WebSocket` class).
- **`ibis-framework[duckdb]==12.0.0`** from `[ask]` — pulled transitively by `nao-core==0.1.8`; zero `import ibis` sites in `src/` or `tests/`.

Plus refresh **`nao-core 0.1.8 → 0.1.11`** — the only direct-dep bump worth chasing this cycle (the other available patch-level bumps — `pydantic`, `typer`, `dbt-core`, `dagster*` — are not user-visible).

Net change: 3-4 lines of `pyproject.toml`, 2 rows of `docs/reference/dependencies.md` (the rows go away rather than getting "redundant" notes), and `uv.lock` regenerates.

### OSI semantic-layer scaffolding (headline) ([#28][])

Open Semantic Interchange (OSI) v0.1.1 (current spec, dated 2025-12-11) is a vendor-neutral YAML for semantic-layer constructs — datasets, metrics, dimensions, relationships. Founding partners include dbt Labs, Snowflake, Databricks, ThoughtSpot, BlackRock, RelationalAI. Tycoon ships a thin scaffolder: dbt MetricFlow stays the source of truth, OSI is a portable *published view* downstream tools can consume directly.

> Note: a previous draft of this doc claimed OSI v1.0 was finalized 2026-01-27. That was wrong — the current spec is v0.1.1 dated 2025-12-11. We'll bump tycoon's vendored schema when the consortium ships v1.0.

**Resolved design** (worked through the open questions in [`docs/proposals/v0.1.6-and-v0.1.7-open-questions.md`](../proposals/v0.1.6-and-v0.1.7-open-questions.md)):

| # | Question | Decision |
|---|---|---|
| Q1 | Command shape | New `tycoon semantics` namespace (parallels `tycoon profiles` / `tycoon register`) |
| Q2 | File layout | Project-level `dbt_project/semantic/osi.yaml`, one file with all metrics |
| Q3 | Auto-aggregation | **Conservative** — scaffold datasets + dimensional fields, leave metrics empty. OSI metrics are SQL expressions (`SUM(orders.amount)`) — tycoon can't reasonably guess intent. Numeric columns get fields with no `dimension:` attribute (measure-eligible bases for user-defined metrics). |
| Q4 | Sentinel + `--force` | Yes — `# @generated by tycoon semantics`, same pattern as `data analyze` |
| Q5 | Schema validation | `tycoon semantics doctor` subcommand (CI-friendly hard exit) plus row in `tycoon doctor` |
| Q6 | Auto-trigger | Off by default in v0.1.6. Opt-in via `transform.auto_osi_scaffold: true`. Default flips on in v0.1.7+ |

Path B (Nao consumes OSI) is deferred until dbt-core ships native OSI export. Path C (Rill `metrics_views` from the OSI source) is deferred until Rill joins the OSI consortium.

### Fivetran metadata read-out ([#26][])

When a user picks `Fivetran` in `tycoon init`, today nothing happens beyond setting `ingestion_managed: false` — tycoon's observability surfaces (`tycoon data status`, `tycoon data history`, the `_tycoon_dlt_usage` Rill dashboard) go dark for the ingestion layer. v0.1.6 adds a read-only Fivetran [Metadata API](https://fivetran.com/docs/rest-api/metadata) client. Tycoon still doesn't *run* Fivetran — it pulls connector + sync metadata and mirrors it into `.tycoon/metadata.duckdb` so the existing surfaces light up.

Scope:

- New typed `stack.ingestion_metadata` block in `tycoon.yml`:
  ```yaml
  ingestion: fivetran
  ingestion_managed: false
  ingestion_metadata:
    api_key: ${FIVETRAN_API_KEY}
    api_secret: ${FIVETRAN_API_SECRET}
    group_id: my_group
  ```
- New `tycoon data fivetran sync` and `tycoon data fivetran list` commands.
- Pulls connector list (id, name, schema, service, paused, sync/setup/update state, succeeded_at, failed_at). Each sync writes one snapshot row per connector keyed on `(connector_id, captured_at)`.
- New `fivetran_connectors` table in `.tycoon/metadata.duckdb`. `tycoon data status` includes a Fivetran panel; `tycoon doctor` validates API credentials + group_id.

**Scope simplification from #26's original spec:** dropped the proposed `[fivetran]` install extra. `httpx` is already a base dependency (it's used by the LM Studio probe + auto-detect logic), so a `[fivetran]` extra would have added zero install footprint and just confused users about what gets installed. The Fivetran code paths gate on `stack.ingestion = fivetran` instead, same way Airbyte / Meltano / "external" already do.

Out of scope: schema-level column metadata (let dbt source freshness handle that side), triggering syncs from tycoon (read-only only), Airbyte Cloud / Stitch / Meltano Cloud (separate issues if/when).

Originally targeted v0.2.x; pulled into v0.1.6 since the surface is small and overlaps directly with the dbt-shop persona served by the profile-handling work below.

### First-class dbt profile handling ([#27][])

Profile resolution today is split between `tycoon register` (correct) and `dbt.py` / `orchestration/resources.py` (hardcoded to the tycoon dev tree, broken for installed users). No `data transform` / `data sync` / `data analyze` command accepts `--profile` / `--profiles-dir` / `--target` flags, and there's no way to ask tycoon "which profile will you use?" without opening the YAML.

Scope for the cycle:

- Centralize resolution in a new `src/tycoon/dbt_profiles.py` helper. Order matches dbt: CLI flag → `tycoon.yml` → `<dbt_project_dir>/profiles.yml` → `$DBT_PROFILES_DIR` → `~/.dbt/profiles.yml`.
- Add `--profile` / `--profiles-dir` / `--target` to every dbt-touching command (`data transform`, `data sync`, `data analyze`, plus the Dagster resource and the FastAPI dbt route).
- New top-level `tycoon profiles` namespace: `list` (discovered profiles + targets + adapters), `show NAME` (pretty-print, secrets redacted), `doctor` (validates active profile + adapter matches `stack.warehouse`).
- Wire `profiles doctor` into `tycoon doctor` as a non-fatal row.

Defers `tycoon profiles add/set` (write-side editing) and encrypted secret handling — read-only for v1.
