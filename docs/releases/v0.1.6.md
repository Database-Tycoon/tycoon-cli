### OSI semantic-layer scaffolding (headline) ([#28][])

Open Semantic Interchange (OSI) v0.1.1 (current spec, dated 2025-12-11) is a vendor-neutral YAML for semantic-layer constructs â€” datasets, metrics, dimensions, relationships. Founding partners include dbt Labs, Snowflake, Databricks, ThoughtSpot, BlackRock, RelationalAI. Tycoon ships a thin scaffolder: dbt MetricFlow stays the source of truth, OSI is a portable *published view* downstream tools can consume directly.

> Note: a previous draft of this doc claimed OSI v1.0 was finalized 2026-01-27. That was wrong â€” the current spec is v0.1.1 dated 2025-12-11. We'll bump tycoon's vendored schema when the consortium ships v1.0.

**Resolved design** (worked through the open questions in [`docs/proposals/v0.1.6-and-v0.1.7-open-questions.md`](../proposals/v0.1.6-and-v0.1.7-open-questions.md)):

| # | Question | Decision |
|---|---|---|
| Q1 | Command shape | New `tycoon semantics` namespace (parallels `tycoon profiles` / `tycoon register`) |
| Q2 | File layout | Project-level `dbt_project/semantic/osi.yaml`, one file with all metrics |
| Q3 | Auto-aggregation | **Conservative** â€” scaffold datasets + dimensional fields, leave metrics empty. OSI metrics are SQL expressions (`SUM(orders.amount)`) â€” tycoon can't reasonably guess intent. Numeric columns get fields with no `dimension:` attribute (measure-eligible bases for user-defined metrics). |
| Q4 | Sentinel + `--force` | Yes â€” `# @generated by tycoon semantics`, same pattern as `data analyze` |
| Q5 | Schema validation | `tycoon semantics doctor` subcommand (CI-friendly hard exit) plus row in `tycoon doctor` |
| Q6 | Auto-trigger | Off by default in v0.1.6. Opt-in via `transform.auto_osi_scaffold: true`. Default flips on in v0.1.7+ |

Path B (Nao consumes OSI) is deferred until dbt-core ships native OSI export. Path C (Rill `metrics_views` from the OSI source) is deferred until Rill joins the OSI consortium.

### Fivetran metadata read-out ([#26][])

When a user picks `Fivetran` in `tycoon init`, today nothing happens beyond setting `ingestion_managed: false` â€” tycoon's observability surfaces (`tycoon data status`, `tycoon data history`, the `_tycoon_dlt_usage` Rill dashboard) go dark for the ingestion layer. v0.1.6 adds a read-only Fivetran [Metadata API](https://fivetran.com/docs/rest-api/metadata) client. Tycoon still doesn't *run* Fivetran â€” it pulls connector + sync metadata and mirrors it into `.tycoon/metadata.duckdb` so the existing surfaces light up.

Scope:

- New typed `stack.ingestion_metadata` block in `tycoon.yml`:
  ```yaml
  ingestion: fivetran
  ingestion_managed: false
  ingestion_metadata:
    api_key: ${FIVETRAN_API_KEY}
    api_secret: ${FIVETRAN_API_SECRET}
    group_id: my_group
  ```
- New `tycoon data fivetran sync` and `tycoon data fivetran list` commands.
- Pulls connector list (id, name, schema, service, paused, sync/setup/update state, succeeded_at, failed_at). Each sync writes one snapshot row per connector keyed on `(connector_id, captured_at)`.
- New `fivetran_connectors` table in `.tycoon/metadata.duckdb`. `tycoon data status` includes a Fivetran panel; `tycoon doctor` validates API credentials + group_id.

**Scope simplification from #26's original spec:** dropped the proposed `[fivetran]` install extra. `httpx` is already a base dependency (it's used by the LM Studio probe + auto-detect logic), so a `[fivetran]` extra would have added zero install footprint and just confused users about what gets installed. The Fivetran code paths gate on `stack.ingestion = fivetran` instead, same way Airbyte / Meltano / "external" already do.

Out of scope: schema-level column metadata (let dbt source freshness handle that side), triggering syncs from tycoon (read-only only), Airbyte Cloud / Stitch / Meltano Cloud (separate issues if/when).

Originally targeted v0.2.x; pulled into v0.1.6 since the surface is small and overlaps directly with the dbt-shop persona served by the profile-handling work below.

### First-class dbt profile handling ([#27][])

Profile resolution today is split between `tycoon register` (correct) and `dbt.py` / `orchestration/resources.py` (hardcoded to the tycoon dev tree, broken for installed users). No `data transform` / `data sync` / `data analyze` command accepts `--profile` / `--profiles-dir` / `--target` flags, and there's no way to ask tycoon "which profile will you use?" without opening the YAML.

Scope for the cycle:

- Centralize resolution in a new `src/tycoon/dbt_profiles.py` helper. Order matches dbt: CLI flag â†’ `tycoon.yml` â†’ `<dbt_project_dir>/profiles.yml` â†’ `$DBT_PROFILES_DIR` â†’ `~/.dbt/profiles.yml`.
- Add `--profile` / `--profiles-dir` / `--target` to every dbt-touching command (`data transform`, `data sync`, `data analyze`, plus the Dagster resource and the FastAPI dbt route).
- New top-level `tycoon profiles` namespace: `list` (discovered profiles + targets + adapters), `show NAME` (pretty-print, secrets redacted), `doctor` (validates active profile + adapter matches `stack.warehouse`).
- Wire `profiles doctor` into `tycoon doctor` as a non-fatal row.

Defers `tycoon profiles add/set` (write-side editing) and encrypted secret handling â€” read-only for v1.

### `tycoon ask init` â€” standalone Nao bootstrap ([#37][])

Late-cycle pickup, paired with `register dbt --create`. Same shape, different surface: a recovery command for projects whose `tycoon.yml` is wired up but whose `.tycoon/nao/nao_config.yaml` isn't.

`tycoon ask init` is idempotent project-bootstrap â€” reads `ask.llm` from `tycoon.yml`, writes the Nao config + refreshes `AGENTS.md`, no prompts. Three concrete cases this fixes:

1. **Hand-edited `tycoon.yml`** â€” switching `ask.llm.provider` from `lm-studio` to `ollama` (or changing the `base_url`) used to require re-running `tycoon register llm`, which forced a re-prompt for things the user didn't want to change. `ask init` reads the YAML as-is and writes the matching Nao config.
2. **Cloned project from a teammate** â€” `tycoon.yml` ships in git, but `.tycoon/nao/` is gitignored (it carries chat history, sample row previews, sometimes PII). Cloning leaves no Nao config locally. Pre-v0.1.6 the only path was `tycoon ask chat`, which auto-inits but immediately tries to launch the chat UI â€” clunky if you just want to verify the install works. `ask init` is the explicit, no-side-effects bootstrap.
3. **`register llm` half-succeeded** â€” `tycoon.yml` got written but the chained `setup_ask_stack()` raised mid-call (e.g. transient FS error). Re-running `register llm` worked but re-prompted; `ask init` is the targeted fix.

History note: `tycoon ask init` was removed in v0.1.5 because it had been a confusing alias for `register llm` (it took LLM provider arguments and prompted). The v0.1.6 re-introduction is a different contract: no LLM args, no prompts, takes whatever's in `tycoon.yml` as gospel. The `_STALE_SUBSTRINGS` sentinel in `tests/test_cli_surface.py` was updated to allow the re-added surface.

Implementation: `setup_ask_stack()` was refactored to delegate to a new `_init_nao_project()` helper. All three init paths (`tycoon init` wizard, `register llm`, the new `ask init`) call the same body â€” no duplication, no drift risk.

### `register dbt --create` â€” recovery path for skipped dbt setup ([#34][])

Late-cycle pickup. Caught while walking through the v0.1.6 demo flow: if a user picks **Skip** on the dbt prompt during `tycoon init`, the only way to add dbt later was to hand-write a `dbt_project.yml` + `profiles.yml` somewhere on disk and then `tycoon register dbt <path>`. The scaffolder that the init wizard uses (`_scaffold_dbt_project` in `src/tycoon/scaffolding/templates.py`) was never reachable from the post-init CLI.

Fix is small and surgical: a `--create` flag on `tycoon register dbt` that calls the same scaffolder, defaults to a `../<project>-dbt` sibling path (matching init's "Create new" option), reads warehouse + raw paths from the active `tycoon.yml`, and falls through to the existing register flow once the dbt project is on disk.

Limitations stay tight: DuckDB / MotherDuck only (the underlying scaffolder doesn't write Snowflake / BigQuery profiles), refuses to overwrite an existing `dbt_project.yml`, marks `stack.transformation_managed: true`. Cloud warehouses still go through plain `tycoon register dbt <path>` after hand-authoring the profile â€” same as before.
