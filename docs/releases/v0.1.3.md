## Planned scope

These are the seven themes targeted for v0.1.3. Promote each into the
"Added" / "Changed" / "Fixed" sections above as it lands. Items that
don't ship before the release date should be moved to the next "What's
next" section rather than partially-described here.

### 1. Template parameterization

Templates like `github-analytics` and `weather-station` hard-code
placeholder tokens (`{owner}/{repo}`, `{station_id}`, `{office}`) in
their URL patterns today, which means their e2e tests only verify
`init` ‚Äî not an actual fetch. v0.1.3 adds a mechanism for templates to
declare named parameters that get prompted at `sources add` or
`sources run` time (extending the existing `${ENV_VAR}` interpolation
with `{{template.param_name}}`-style placeholders or similar).

Unblocks: the github-analytics and weather-station e2e tests can run
full ingestion pipelines.

### 2. Snowflake / BigQuery warehouse alignment

v0.1.2 extended the v0.1.1 warehouse-alignment check from DuckDB-only
to MotherDuck. v0.1.3 extends it to Snowflake and BigQuery. The
tradeoff is that those dbt profiles aren't structurally comparable by
a single "path" field ‚Äî Snowflake has account/user/password, BigQuery
has project/keyfile. Probably the alignment check reduces to: does
the dbt profile's target warehouse type match `stack.warehouse`? And
for Snowflake, does `account` match?

### 3. `dbt build` step in e2e tests

The csv-import e2e test currently runs `init ‚Üí sources run` and
asserts on raw DuckDB row counts. v0.1.3 extends it to also run
`tycoon data transform run` and assert on transformed output.
Prerequisite: templates need at least one dbt model shipped.
`csv-import` is the first candidate ‚Äî a single `stg_files` staging
model passing-through the ingested data is enough.

### 4. dlt trace.json enrichment (observability v2a)

v0.1.2's dlt observability captured `_dlt_loads` + per-table row
counts. Trace-level detail (byte sizes, per-job durations, error
messages) lives in `~/.dlt/pipelines/<name>/trace.json`. v0.1.3 adds
a `capture_dlt_trace` helper that parses those files and extends
`dlt_runs` (or a new `dlt_trace` table) with byte sizes and per-step
timings. Surfaces in the Rill dashboard + `tycoon data history show`.

### 5. dbt manifest.json schema-diff (observability v2b)

The current `dbt_runs` / `dbt_nodes` capture what happened per
invocation but doesn't catch "this model's SQL hash changed between
runs" or "a column was added / renamed / dropped." v0.1.3 snapshots
`target/manifest.json` alongside `run_results.json` and computes a
diff against the previous snapshot: new models, removed models,
renamed columns, SQL-hash changes. Surfaces as a new
`dbt_schema_changes` table + a drilldown in `tycoon data history show`.

### 6. One-command MotherDuck + Nao + LM Studio setup

Follow-through on issue
[#7](https://github.com/Database-Tycoon/tycoon-cli/issues/7). A
single command (`tycoon register stack` or similar) walks through:
(a) MotherDuck auth (OAuth or token), (b) Nao init + schema sync,
(c) LM Studio (or any local OpenAI-compatible endpoint) detection +
wiring. Today each of these is a separate 2‚Äì5 step process and
ordering matters.

### 7. `tycoon data sync` ‚Äî cloud ‚Üî local DuckDB snapshots

Follow-through on issue
[#12](https://github.com/Database-Tycoon/tycoon-cli/issues/12). A
new subcommand that snapshots a MotherDuck warehouse to a local
DuckDB file (for offline analysis / AI training / air-gapped
demos), and conversely uploads a local DuckDB to MotherDuck. Uses
DuckDB's native `ATTACH 'md:...'` + `COPY FROM` under the hood;
tycoon's job is just the auth + path resolution + naming conventions.

## Test + CI goals for v0.1.3

- **Coverage floor ratchet**: raise from 60% (v0.1.2 floor) to
  **62‚Äì63%** if feasible. Each new capture helper + parameterization
  path should ship with tests that push the overall number up.
- **Full e2e gate on every PR**: once `dbt build` runs in csv-import's
  e2e, promote any future template that acquires a full offline
  pipeline into the `offline_e2e` marker too.
- **Dependabot**: a follow-up to v0.1.2's dependency bumps ‚Äî configure
  `.github/dependabot.yml` so Python and Actions deps get automated PRs
  we can review at a glance instead of bundling them into release
  cycles by hand.
