# Tycoon v0.1.4

_Released: TBD_

## Headline

v0.1.4 is the follow-through release for the two XL items that slipped from
v0.1.3 ((`tycoon data sync` and one-command MotherDuck/Nao/LM Studio setup),
plus housekeeping that was visible-but-deferred during v0.1.3.

## Carried forward from v0.1.3

### 1. `tycoon data sync` — cloud → local DuckDB snapshots ✅ landed

Issue [#12](https://github.com/Database-Tycoon/tycoon-cli/issues/12).
Snapshot one or more DuckDB-attachable sources into a single local file:

```bash
tycoon data sync --from md:my_catalog --to ./data/local.duckdb
tycoon data sync --from md:dlt_prod --from md:dbt_prod --to ./snap.duckdb
tycoon data sync                                   # uses tycoon.yml's sync block
tycoon data sync --schema mart --tables 'dim_*'    # filtered
tycoon data sync --mode skip-existing              # first-run-only seeding
```

Implementation lives in `src/tycoon/sync.py` (core ATTACH/COPY logic,
result types) + `src/tycoon/commands/sync_cmd.py` (CLI surface).
`SyncSourceSpec` and `SyncConfig` pydantic models live in
`src/tycoon/project.py` so a `sync:` block can ship in `tycoon.yml`:

```yaml
sync:
  to: data/local_snapshot.duckdb
  sources:
    - from: md:dogfood_dlt_prod
      schemas: ['*']
    - from: md:dogfood_dbt_prod
      schemas: ['mart']
```

**v1 deliberate non-goals** (per the issue): no reverse sync (local →
cloud), no incremental, no catalog rename/remap, only `md:` + local
DuckDB sources for now. The shape leaves room to add Postgres / Snowflake
/ BigQuery as their respective DuckDB ATTACH stories settle.

**Tests.** `tests/test_sync.py` covers the core function (replace /
append / skip-existing modes, schema + table glob filtering, system
schema exclusion, multi-source merging, parent-dir auto-creation, mode
validation) and the CLI surface (no-source error path, no-destination
error path, unknown-mode error path, end-to-end success, schema filter,
config-block fallback). 16 new tests; all pass.

A reverse-sync (local → cloud) story remains explicitly out of scope —
keeps the model simple and prevents accidental prod mutation. Revisit
when there's a clear incremental story; for v1 the snapshot is a
baseline, not a replica.

### 2. One-command MotherDuck + Nao + LM Studio setup ✅ partially landed

Issue [#7](https://github.com/Database-Tycoon/tycoon-cli/issues/7).
Shipping in v0.1.4: the four sub-asks that don't require redesigning
the init wizard. The remaining two (init-wizard LLM prompt + smart
include defaults) are deferred — the wizard is being redesigned in
v0.2.0 (#14), so adding LLM-prompting now would just be churn.

**§4 — Pre-create every dir `nao sync` walks unconditionally.**
`tycoon ask init` now creates all eight dirs (`databases/`, `queries/`,
`docs/`, `semantics/`, `repos/`, `agent/{tools,mcps,skills}`)
up front, so `nao sync` runs cleanly without a separate `nao init`
step.

**§5 — LM Studio first-class.** New `tycoon ask init --llm <provider>`
flag with shortcuts for `lm-studio` / `ollama` / `openai` / `anthropic`
/ `gemini` / `mistral`. `lm-studio` is the marquee path — it expands
to an OpenAI-compatible nao config pointed at `http://localhost:1234/v1`
with a placeholder `api_key: lm-studio` so users don't have to discover
that "openai + custom base_url" is the LM Studio idiom. `LLMConfig`
gains a `base_url` field.

**§6 — `tycoon ask doctor` health check.** New subcommand validating
the four common breakage modes from the issue: missing `nao_config.yaml`,
missing required directories, missing MotherDuck auth, unreachable LM
Studio endpoint. Renders a `status_table` with OK / WARN / FAIL per
panel. WARN on missing MD token (could be on OAuth — matches `tycoon
doctor` from v0.1.2 #3); FAIL on unreachable LM Studio endpoint when
the provider is configured. Exits non-zero on any FAIL so it's
CI-friendly.

**§7 — Auto-generated `.tycoon/nao/.gitignore`.** Issue's PII concern
(row previews in `databases/*/preview.md`) addressed via
auto-generated `.tycoon/nao/.gitignore` ignoring `databases/`,
`repos/`, and `db.sqlite*`. Sentinel-detect own file like AGENTS.md;
preserve user-authored .gitignore.

**Tests.** `tests/test_nao.py::{TestNaoDirScaffolding, TestNaoGitignore,
TestLmStudioPreset}` — 10 new tests covering the helpers + LM Studio
expansion + edge cases. `tests/test_ask_context.py::{TestAskDoctor,
TestAskInitLlmFlag}` — 6 new tests covering the CLI surfaces.

### Deferred to v0.2.0 init redesign (#14)

- §3: smart include_schemas defaults (introspect dbt for `mart` schema,
  prompt otherwise)
- §5b: init-wizard LLM provider prompt (the current init wizard
  doesn't prompt for LLM at all; adding it overlaps with #14's full
  init redesign)

### 3. Rill 0.86 DuckLake live-connector migration — revisit

Probed end-to-end during v0.1.3 and **deferred**: SQLite-backed DuckLake
catalogs hold an OS-level exclusive lock that breaks tycoon's "Rill running
while pipelines write" workflow, which the existing Parquet bridge handles
fine. Two paths to revisit:

- Wait for Rill to ship shared-lock semantics on the DuckLake connector.
- Adopt a Postgres-backed DuckLake catalog story (would break tycoon's
  local-first positioning unless Postgres is opt-in).

If neither materializes during v0.1.4, this becomes a follow-up.

A smaller piece of the same theme — **adding `connector:` to scaffolded
metrics-view YAML and `output:` to scaffolded model YAML** — is forward-
compatible with Rill 0.86 and is on the v0.1.4 candidate list. Needs
testing against existing v0.1.x-scaffolded projects to make sure the new
fields don't change behavior.

### 5. nyc-transit live e2e test ✅ landed

Issue [#17](https://github.com/Database-Tycoon/tycoon-cli/issues/17),
filed during v0.1.3 pre-release validation. **Root cause:** the three
legacy NYC pipeline modules (`nyc_dot_pipeline`, `mta_pipeline`,
`mta_bus_speeds_pipeline`) were importing the global
`tycoon.config.config` singleton and using `config.raw_db` to set
dlt's destination, instead of using the `raw_db_path` that the runner
threads through for every non-legacy source. That worked in a real CLI
process where global config and the active project line up — and broke
the e2e test, which uses `monkeypatch.setattr(sources_mod, "config",
cfg)` to point at a tmp_path. The rebind hits `commands.sources.config`
but not the legacy module's reference; dlt wrote to the singleton's
`data/raw.duckdb` default, not the project's
`data/nyc_open_data_raw.duckdb`, and the test's existence assertion
correctly failed.

**Fix:** `_run_legacy` now passes `raw_db_path` through to each
pipeline's `run_pipeline()`. The three pipeline modules dropped their
`from tycoon.config import config` import and use `raw_db_path` directly
for both the dlt destination and the parent-dir mkdir. Test runs in
2.57s.

### 6. Hygiene: `_tycoon_*` rill artifacts in CLI source repo

Surfaced during v0.1.3 testing. The CLI source repo's own `rill/`
directory accumulates `_tycoon_dlt_*.yaml` artifacts when tycoon is
invoked against itself during tests. The repo's `.gitignore` already
has a "this is the CLI source repo, not a tycoon project" block —
extend it to cover the auto-generated rill artifacts.

## Candidates for further consideration

- **`tycoon ask context` polish.** Add a `--json` mode for programmatic
  callers; let `--include-dbt` filter to a specific schema or table;
  consider an `mcp` mode that exposes the context as MCP tools.
- **`AGENTS.md` per-template flavor.** The current generator emits the
  same body regardless of template; templates with strong domain rules
  (e.g. `nyc-transit`) could ship template-specific RULES.md content.
- **Observability v3.** Capture the chat history written by Nao
  (`.tycoon/nao/db.sqlite`) as additional rows in
  `.tycoon/metadata.duckdb` so `tycoon data history` can show recent
  agent questions alongside dlt and dbt runs.
