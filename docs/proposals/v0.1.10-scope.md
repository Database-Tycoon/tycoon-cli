# v0.1.10 scope & goals

_Status: scoped 2026-06-13 after v0.1.9 shipped to PyPI (2026-06-12). No implementation landed yet — this is the goal/plan file for the cycle. Milestone: GH `v0.1.10` (#5); Jira `target-v0.1.10`._

## Goal

**Lock it down, plus a few nice-to-haves.** v0.1.10 is primarily a **security hardening pass** — closing the Medium/Low findings from the 2026-06-10 security review (the two High findings shipped in v0.1.9). Alongside the hardening, a small set of features and the contributor/community groundwork ride along.

One-liner: **7 security fixes + 6 small features, with 3 bigger items parked until their dependencies land.**

## Simple breakdown

### 🔒 Security (7) — the heart of the release
Locking down what a shared/malicious project could exploit:

- **#62 / #65** — `tycoon.yml` config can't leak env secrets (`${VAR}` exfiltration) or inject paths/identifiers
- **#63** — local web servers don't expose to the network (bind `127.0.0.1`, Host/CORS guards)
- **#66** — Quack token file locked down (perms + token-shape validation)
- **#67** — scheduled-run unit files can't be hijacked (systemd escaping)
- **#64 / #68** — CI/build supply chain hardened (SHA-pin actions, Dependabot) + secret-scanning/SAST + `SECURITY.md`

### ✨ Features & polish (6)

- **#35** — `ask refine`: AI refines generated dbt models / Rill dashboards
- **#36** — `tycoon serve / ps / kill`: start/stop/list background services
- **#47** — gate-publish-on-checks recipe: block publishing data that fails quality checks
- **#31** — layer-granular backup (`backup.layers`) — ⚠️ blocked on the deferred DuckLake backup track; may slip
- **#53 / #54** — contributor docs (CONTRIBUTING + templates) and community setup (Discussions + good-first-issue) — owned by Karli

### ⏳ Later (3) — explicitly NOT this release

- **#14** — redesign `tycoon init` → next major (v0.2.0)
- **#43** — DuckDB 2.0 support → waiting on the engine (target Q4 2026)
- **#58** — dbt Core v2 / Fusion → blocked on upstream (dbt-duckdb v2, dagster-dbt Fusion)

## Recommended start order

1. **#62 + #65** first — they share the same root cause (untrusted `tycoon.yml`) and finish the malicious-config threat model that v0.1.9's two High fixes started.
2. **#63 / #66 / #67** — the mechanical, contained hardening fixes.
3. **#64 / #68** — CI / supply-chain / process.
4. Features (#35, #36, #47) and the community docs (#53, #54) as capacity allows; **#31 is blocked** and should only be picked up if the DuckLake backup track lands first.

### Scope risk

13 issues is a large, mixed milestone (7 security + 6 feature/community, one blocked). If a tighter cut is wanted, the natural trim is **security-only v0.1.10** (the 7) — matching the "hardening pass" identity — pushing the features to v0.1.11; optionally keep the two low-effort community docs (#53, #54).
