# v0.1.8 scope — Quack, DuckLake backup, prod ergonomics

_Status: draft proposal, 2026-05-25. To be resolved before implementation starts._

## Context: where we are after v0.1.7

v0.1.7 (released 2026-05-25) shipped the **layered data model** —
sources / staging / intermediate / marts as first-class — plus the
snapshot harness for Rich-rendered CLI output and non-interactive
`tycoon data sources add`. Tycoon now understands the *shape* of a
modern analytics warehouse and surfaces it consistently across
`data status`, `doctor`, `semantics scaffold`, and `data history`.

What it doesn't yet do:

- **Multi-process concurrency on the local DuckDB.** Every consumer
  (CLI, Nao, dbt, Rill) opens the warehouse file in-process. Only
  one writer at a time. Switching between `ask chat` and a running
  `dbt build` means hand-stopping one.
- **Cloud-bucket backup.** The DuckLake design from v0.1.7's open
  questions doc was written but deferred. Right now `tycoon data
  sync` does cloud → local snapshots only; there's no `tycoon data
  backup` going the other way.
- **Operate-in-prod ergonomics.** A scheduled `tycoon data run-all`
  with no notification surface is invisible until someone checks the
  terminal. No scheduling primitive beyond "configure your own
  cron." Failure modes get noticed by the business, not by us.
- **Live-data freshness in `data status`** for Fivetran. v0.1.7's
  Sources panel reads from cache; the design Q3 was live API read
  with write-through cache.

Plus a new arrival: **[DuckDB Quack][quack]** went public 2026-05-12
as a `core_nightly` extension. It turns a local DuckDB into a
multi-client server — exactly the concurrency primitive tycoon is
missing.

[quack]: https://duckdb.org/quack/

The user has flagged Quack as the headline. This doc plans v0.1.8
around it, with backup + ergonomics riding along, and DuckDB 2.0 as
a separately-tracked dependency watch (still rumored Q4 2026).

## Track 1: Quack protocol integration ([#42])

### What's settled

- **Scope: local-only.** Server binds to `localhost`; no
  cross-machine variant. The local-first ethos is preserved.
- **Mechanism: Quack extension on the existing DuckDB.** No
  separate runtime, no daemon binary to manage. The server is
  literally a SQL session running `CALL quack_serve(...)`.
- **Auth: token-based via DuckDB secret.** Generated per project,
  persisted in `tycoon.yml` (probably under a `serve:` block).
- **Opt-in.** v0.1.8 ships Quack as a feature flag; the default
  warehouse path stays in-process. We make Quack discoverable but
  don't break anyone's existing workflow.

### Open questions

#### Q1 — Where does the server live?

Three plausible homes:

**a)** `tycoon serve --quack` — a new top-level subcommand. Pairs
with the existing `tycoon serve` (FastAPI UI) — clean
verb/object split.

**b)** `tycoon data serve` — under the `data` namespace where the
warehouse lives.

**c)** `tycoon start --only quack` — wires into the existing
`ServiceManager` so it's backgrounded the same way Rill is. The
PID lives in `.tycoon/run/pids.json`; `tycoon stop` shuts it down.

Lean: **(c) ServiceManager-backed**, with `tycoon serve --quack`
(option a) as a one-shot foreground convenience for debugging.
Matches the pattern Rill uses today.

#### Q2 — Which clients attach via Quack?

| Client | Today | v0.1.8 plan |
|---|---|---|
| `tycoon data db query` | Opens warehouse file in-process | If Quack server is running, attach as `quack:localhost`; else keep file-based |
| `tycoon ask chat` (Nao) | Reads via Nao's config | Nao gets a Quack-aware config when server is up; falls back to file mode |
| dbt | Opens warehouse via dbt-duckdb profile | **Open question.** dbt-duckdb's adapter may not understand `quack:` URIs yet. Default to file mode; investigate adapter contribution upstream as separate work. |
| Rill | Reads parquet exports (file connector) | Unchanged — Rill doesn't talk to DuckDB live, it reads parquet. Quack has no role here. |
| `tycoon data db shell` | Opens REPL on the file | If server is up, attach via Quack so user shares state with running consumers |

**dbt is the hard one.** If dbt can't use Quack, we have a
gotcha: running `dbt build` while Quack is up means dbt is locked
out of the warehouse (it'll try to open the file and fail). Two
mitigations:

- **Auto-pause Quack during dbt commands.** `tycoon data transform run`
  detects a running Quack server, pauses it, runs dbt, resumes it.
  Same trick we play for `tycoon data sync` skip-on-error today.
- **Document the limitation.** Users running standalone `dbt build`
  outside tycoon get a clear error when the warehouse is busy.

Lean: **auto-pause for `tycoon data transform *` commands.** Matches
the existing single-writer mental model; users don't have to think
about it.

#### Q3 — Token lifecycle

Three plausible:

**a)** Generated once at `tycoon init` (or first `tycoon serve --quack`)
and persisted forever in `.tycoon/secrets.yml` (gitignored).

**b)** Regenerated per `tycoon serve --quack` invocation. Server
prints the token; clients have to be told.

**c)** Time-limited — token rotates every N hours.

Lean: **(a) persisted once, rotate via `tycoon serve --quack --rotate-token`.**
Lowest friction. Local-only access scope makes rotation low-stakes.

#### Q4 — How do we tell users Quack is the recommended path?

`tycoon doctor` gets a new row:

- If `[quack]` extra is installed and server isn't running →
  "Quack server: not running. Start with `tycoon start --only quack`
  to enable multi-process access to your warehouse."
- If server is running → "Quack server: running on localhost:9494."
- If consumer config (Nao's `nao_config.yaml`) points at the file
  path while server is up → suggest the fix.

#### Q5 — Extension stability

Per #42's open question: Quack is in `core_nightly` as of
2026-05-12. v0.1.8 ships within the next ~3 weeks; we'll likely
still be on nightly at release.

Implications:

- **Install instructions assume `core_nightly`.** Users running an
  older DuckDB without the nightly extension repo see an actionable
  error from `tycoon doctor`.
- **Feature-flag.** `tycoon serve --quack` errors out with a clear
  message if the extension isn't loaded. We don't try to be clever
  with installation — `INSTALL quack FROM core_nightly` is one
  copy-paste away.
- **Graduation path documented.** When Quack moves to stable `core`
  (probably with DuckDB 2.0 per #43), the install instructions
  simplify; the rest of tycoon's surface stays the same.

### Proposed scope (v1 ship)

- [ ] `tycoon serve --quack [--port 9494] [--rotate-token]` — start a
      Quack server bound to the active project's warehouse
- [ ] `tycoon start --only quack` integration via ServiceManager
- [ ] `tycoon stop` knows about Quack
- [ ] `serve:` block in `tycoon.yml` (port, token reference)
- [ ] `.tycoon/secrets.yml` carrying the generated token (gitignored)
- [ ] `tycoon data db query` / `tycoon data db shell` Quack-aware
- [ ] `tycoon ask chat` config generator (writes nao_config.yaml with
      Quack attachment when server is configured)
- [ ] Auto-pause Quack during `tycoon data transform *` commands
- [ ] `tycoon doctor` Quack row (4 states)
- [ ] `[quack]` install extra in pyproject — empty for now, exists
      so we can swap to a `duckdb-quack` package if one ships
- [ ] `docs/recipes/multi-process-with-quack.md` — recipe explaining
      the new flow, including the dbt auto-pause behavior

### Out of scope (deferred)

- **Cross-machine Quack.** Local-only for v1; cross-machine is its
  own design problem (firewall, auth scope, network topology).
- **Quack proxy for MotherDuck.** MotherDuck is already a hosted
  multi-client surface; Quack adds nothing there.
- **dbt-duckdb upstream contribution** for native Quack URI support.
  Worth pursuing but not on the v0.1.8 critical path.

## Track 2: DuckLake cloud-bucket backup (deferred from v0.1.7)

The full design is in `docs/proposals/v0.1.6-and-v0.1.7-open-questions.md`
§"v0.1.7 — Cloud bucket checkpoint via DuckLake" (lines 213-440).
That doc is canonical; this section is the v0.1.8 implementation plan
on top of it.

### What's settled (carried from the v0.1.7 design)

- **Backup mode**, not bidirectional sync. Single canonical state on
  the bucket; local is the source of truth.
- **Layers backed up**: warehouse + raw + metadata DBs. Default
  includes raw; opt out via `backup.skip_raw: true`.
- **Trigger**: hybrid — auto-sync after pipeline ops (default-on);
  manual `tycoon data backup` available; opt-out via
  `backup.auto_sync: false`. Smart-skip via mtime.
- **Restore UX**: smart default (download to canonical if absent,
  else to siblings + require `tycoon data backup promote`). `--force`
  and `--safe` overrides.
- **Provider abstraction**: DuckLake's httpfs handles S3 / GCS /
  Azure / R2 via uniform URI schemes. Auth via env vars.
- **Catalog placement**: inline at `<bucket>/_lake_catalog.duckdb`.
  Bucket is self-contained.

### Open questions (carried from v0.1.7 design, still unresolved)

These need answers before implementation starts. The v0.1.7 design
doc enumerates 8 open questions; the four below are the ones that
materially shape v0.1.8's API:

#### Q1 — Catalog DB choice

DuckDB-on-S3 (single-writer, simple) vs. a hosted Postgres catalog.
Lean **DuckDB-on-S3 for v1** — simpler, no second service. Document
the upgrade path to Postgres when concurrent writers become a real
need.

#### Q2 — Snapshot retention default

`retain_snapshots: all | <N>_days | latest`. Default to `latest`
(safest from a cost perspective). Power users opt up.

#### Q3 — Encryption story

Two layers: bucket-side server encryption (handled by AWS/GCP/etc.,
no tycoon code needed) and at-rest encryption of secrets in
`tycoon.yml`. v0.1.8 documents how to use env vars for the AWS keys;
deeper encryption is its own follow-up.

#### Q4 — `tycoon doctor` integration

- Bucket reachability ("can we list the prefix?")
- Catalog age ("`_lake_catalog.duckdb` last modified Nh ago")
- Local-vs-bucket drift summary

### Proposed scope (v1 ship)

- [ ] `BackupConfig` pydantic model in `tycoon.project` —
      `destination`, `auto_sync`, `skip_raw`, `retain_snapshots`
- [ ] `tycoon register backup --to s3://...` — initialize the bucket,
      write `backup:` block to `tycoon.yml`
- [ ] `tycoon data backup` — manual push of local DuckDB files to
      the lake via DuckLake httpfs
- [ ] `tycoon data backup pull` — restore (smart-default behavior)
- [ ] `tycoon data backup promote` — swap restored siblings into
      canonical paths
- [ ] `tycoon data backup status` — local/lake state diff
- [ ] Auto-sync hooks in `tycoon data sources run` /
      `tycoon data transform *` (smart-skip via mtime)
- [ ] `tycoon doctor` backup row (3 states: not configured, healthy,
      drift detected)
- [ ] `tests/test_backup.py` — mock-bucket harness; integration with
      moto / fake-s3 for an end-to-end test
- [ ] `docs/commands/data/backup.md`
- [ ] `docs/recipes/cloud-backup.md`

## Track 3: Layer-granular backup config ([#31])

Rides on Track 2. Once `backup.destination` + `backup.skip_raw`
exist, swap `skip_raw: bool` for `layers: list[str]`.

### Settled by issue body

- `backup.skip_raw: true` desugars to `layers: [staging, intermediate, marts]`.
- Restore auto-promotes upstream dependencies (matches dbt's `+marts`
  selector semantics).
- `tycoon doctor` validates the layer list against actual warehouse
  contents.

### Proposed scope

- [ ] `BackupConfig.layers: list[Layer]` field (using the v0.1.7
      `Layer` enum from `tycoon.layers`)
- [ ] Backwards-compat path: `skip_raw: true` rewrites to
      `layers: [staging, intermediate, marts]` on read
- [ ] `tycoon doctor` row catches typos in the layer list
- [ ] Restore auto-promote logic (resolve lineage upward through
      `dbt_nodes` to materialize prerequisites)
- [ ] Tests covering the four cases enumerated in the issue body

## Track 4: Operate-in-prod ergonomics

### #46 — Slack / webhook notifications

Settled by issue body: "An alert that posts to a Slack channel if
our job succeeds or fails." Smallest production-grade gap tycoon has.

Proposed surface:

```yaml
# tycoon.yml
notify:
  webhook: https://hooks.slack.com/services/T.../B.../...
  on: [success, failure]   # or just [failure]
```

- [ ] `NotifyConfig` model
- [ ] Webhook POST helper (httpx, already a base dep) with retry
- [ ] Hook into `tycoon data run-all`, `tycoon data sources run`,
      `tycoon data transform run`
- [ ] `tycoon register notify --webhook URL` for one-shot setup
- [ ] `--no-notify` flag on every hooked command
- [ ] Tests with httpx.MockTransport

### #48 — cron / launchd schedule wrapper

Settled by issue body: one tycoon command, platform-native scheduler,
no daemon.

Proposed surface:

```bash
tycoon schedule add daily-refresh --at 04:00 --command "data run-all"
tycoon schedule list
tycoon schedule remove daily-refresh
```

- [ ] Detect platform (`launchd` on macOS, `cron` on Linux)
- [ ] launchd: write `.plist` to `~/Library/LaunchAgents/`, `launchctl load`
- [ ] cron: append a line to crontab via `crontab -l | ... | crontab -`
- [ ] `tycoon.yml` carries no schedule state — the source of truth is
      the OS scheduler. `tycoon schedule list` shells out to find ours
- [ ] Log capture: redirect output to `.tycoon/logs/<schedule>.log`,
      rotate weekly
- [ ] Pairs with #46 — scheduled runs auto-send notifications
- [ ] Tests: dry-run mode that emits the platform-native config
      without actually installing it

## Cross-cutting questions

### CC-1. Is this too big for one cycle?

Counting tracks:

| Track | Estimated effort |
|---|---|
| 1. Quack | ~10 commits, 800-1200 LoC, 25-35 tests |
| 2. DuckLake backup | ~8 commits, 600-900 LoC, 20 tests, 2 docs |
| 3. Layer-granular (#31) | ~3 commits, 200 LoC, 6 tests (depends on Track 2) |
| 4. Notify + schedule (#46, #48) | ~6 commits combined, 400 LoC, 12 tests |
| 5. Fivetran live read (#50) | ~1 commit, 50 LoC, 3 tests |
| 6. DuckDB 2.0 (#43) | 0 LoC (watch only) |

Realistic total: **~28 commits, ~2500 LoC, ~75 tests, 3-4 weeks**.

v0.1.7 was 6 commits / ~2200 LoC / +22 tests over ~3 weeks. This
proposal is roughly 4x the test load. **It is too big as-is.**

### CC-2. Recommended cuts (in order of "cut first")

1. **Defer #48 (schedule wrapper)** to v0.1.9. It's the most
   self-contained item, doesn't block anything, and has the most
   open shape (launchd vs systemd vs Windows still TBD).
2. **Defer #31 (layer-granular backup)** to v0.1.9. Builds on Track
   2; shipping Track 2 alone is the more conservative play. Re-files
   itself naturally once users actually want it.
3. **If still too big**: collapse Track 4 to just #46 (notify),
   defer #48 as above.

### CC-3. The cycle if all the cuts land

| Track | In v0.1.8? |
|---|---|
| 1. Quack (#42) | ✓ headline |
| 2. DuckLake backup (deferred from v0.1.7) | ✓ |
| 3. Layer-granular (#31) | DEFER to v0.1.9 |
| 4a. Notify (#46) | ✓ |
| 4b. Schedule (#48) | DEFER to v0.1.9 |
| 5. Fivetran live read (#50) | ✓ |
| 6. DuckDB 2.0 (#43) | watch only, no code |

That's **~20 commits, ~1600 LoC, ~50 tests, ~3 weeks**. Tracks
closely to v0.1.7's actual scope.

### CC-4. The Quack ↔ backup interaction

Quack ATTACH semantics work fine over DuckLake — per #42, "supports
DuckLake." So a user can have:

1. Local DuckDB warehouse (canonical state)
2. Quack server exposing it to CLI / Nao / dbt
3. Periodic backup to DuckLake on S3

That's the v0.1.8 happy path. Nothing in Track 1 conflicts with
Track 2; they layer cleanly.

### CC-5. The new `tycoon doctor` rows

Counting up: Quack (1), backup (1), layer-granular (#31, 1 if it
ships), notify (1), Fivetran live (1, replacing the existing cache
row). That's potentially 4-5 new doctor rows in one cycle.

The doctor output is getting long. v0.1.8 should add a `--brief`
flag (or make brief the default with `--full` for the existing
verbose mode). File as a follow-up if it slips.

## Implementation hints (for self, when work starts)

Concrete touchpoints — not commitments, just where to start reading.

### Quack (Track 1)

- ServiceManager pattern for backgrounding lives in
  `src/tycoon/services/manager.py` — same model Rill uses today.
  Quack should plug in there; PID lives in `.tycoon/run/pids.json`,
  `tycoon stop` already iterates.
- Auto-pause-during-dbt: analog is `tycoon data sync`'s per-table
  resilience pattern (`SyncResult.skipped`). Wrap the dbt subprocess
  call in a context manager that suspends the Quack server, runs
  dbt, resumes. One helper, used by every `tycoon data transform *`
  command.
- Token storage: new `.tycoon/secrets.yml` file, gitignored. Don't
  put it in `tycoon.yml` proper — that's checked in. Reference
  with `${file:.tycoon/secrets.yml#quack_token}` or similar.
- Install hint when extension is missing: use the v0.1.6 escaping
  pattern (raw strings, `\[quack]` escaped) so Rich doesn't strip
  brackets. The snapshot harness from #41 will catch regressions
  but the writer needs to do it right initially.

### DuckLake backup (Track 2)

- `BackupConfig` pydantic model goes in `src/tycoon/project.py`
  alongside `StackConfig`, `AskConfig`, `SyncConfig`.
- DuckLake httpfs lives in `src/tycoon/ingestion/ducklake_config.py`
  already (used by Rill bridge). Reuse, don't rebuild.
- The mtime-based smart-skip needs to read DuckDB file mtime AND the
  metadata DB's `dbt_runs` table's `started_at` — a backup is needed
  if either has changed since the last bucket write.
- Test harness: `moto` (S3 mock) or a real LocalStack container. The
  pattern Matt Martin used in his AWS-ETL walkthrough is `moto`;
  fewer moving parts than LocalStack.

### Layer-granular backup (Track 3)

- `Layer` enum already exists in `src/tycoon/layers` from v0.1.7.
  Import it; `BackupConfig.layers: list[Layer]` is the field.
- Auto-promote-on-restore: walk the dbt manifest's `depends_on.nodes`
  upward from each `Layer.MART` node to collect prerequisites. Same
  classifier pass that powers `tycoon data history --layer`; reuse.

### Notify + Schedule (Track 4)

- httpx already a base dep — no new install for #46.
- `NotifyConfig` model alongside `BackupConfig`. Hook into a single
  finalizer function that every `tycoon data *` command calls on
  exit; flag-gated by `--no-notify`.
- launchd vs cron: detect via `sys.platform == "darwin"` for
  launchd. Linux falls through to crontab. Windows is out of scope
  for v1 (file as a follow-up if anyone asks).

### Fivetran live read (Track 5)

- Touchpoint: `_render_sources_panel` in `src/tycoon/commands/status.py`.
- The `build_client_from_config` factory already exists in
  `src/tycoon/ingestion/fivetran_client.py`. Wrap the call in
  try/except FivetranAPIError; on failure, fall through to
  `latest_connector_snapshot` with a `warn()` line.

### DuckDB 2.0 (Track 6)

- Just watching. If 2.0 ships during v0.1.8: don't pull it in;
  cut a v0.1.9 immediately after.
