## #57 design recap — managed project-local `.venv` (decided: Option A)

tycoon owns the Python environment instead of inheriting whatever `python3`/`pip` the user has.

- **Single project-local `.venv/`** containing tycoon + dbt + dlt + duckdb. Matches tycoon's current architecture (in-process `dlt`/`duckdb`; dbt resolved at `Path(sys.executable).parent / "dbt"`), so it's the low-churn fit.
- **Correct interpreter, guaranteed:** create via `uv venv --python <X>` where X ∈ `[3.12, 3.14)`. uv **auto-downloads** a python-build-standalone CPython if the machine only has an unsupported one (e.g. 3.14) → zero manual installs.
- **Pin** with a `.python-version` (3.12) in the **project dir** (safe there; only broke CI when placed at the package repo root).
- **`tycoon doctor` interpreter check** (range `[3.12,3.14)`) + `--fix` / `tycoon setup` that rebuilds `.venv`. Today doctor checks no Python version — that's why #55 was invisible until `transform run` failed.
- **Bootstrap:** `uvx database-tycoon init` (or a one-line installer) builds `.venv` with the right Python and installs everything; thereafter the user uses that `.venv`.
- Locate uv via `shutil.which("uv")`; offer to bootstrap uv (standalone installer) if absent. Confirm whether the `uv` wheel exposes a binary-locator API at impl time.
- **Deferred — Option B** (tycoon as a global `uv tool` + a separate project env it shells into): needs in-process `dlt`/`duckdb` refactored to subprocess + adds two-environment confusion. Revisit later.
- Collapse the current 3-way install menu (pip / uv tool / uv add) into one recommended onboarding path in the docs.

## Release-wrap checklist (learned the hard way; verify before declaring v0.1.9 done)

- **All four release surfaces:** PR merged + tag pushed + PyPI live + **GitHub Release page created** (the last was missed in v0.1.7; `publish.yml` does not create it — `gh release create` is manual).
- **Version coherence preflight:** `pyproject.toml` ↔ `src/tycoon/__init__.py` ↔ tag ↔ CHANGELOG date ↔ `docs/releases/v0.1.9.md` `_Released:` line. `publish.yml` fails fast on drift.
- **CHANGELOG + release notes stay in sync as work lands**, not at the end.
- **mkdocs nav:** add `v0.1.9` to the Releases nav; mark the prior release released; suppress this scope doc in `not_in_nav` (it's a draft proposal) so `mkdocs build --strict` stays clean.
- **Branch hygiene:** this branch (`v0.1.9`) is already cut. PR into `main`; tag the merge commit.
