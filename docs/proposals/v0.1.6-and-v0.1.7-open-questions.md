# Open design questions — v0.1.6 (OSI, profiles, Fivetran) + v0.1.7 (Cloud bucket via DuckLake + layer-aware data model)

> Status: **Design discussion in progress.** Not part of the user-facing
> docs site (excluded from nav). Lives in `docs/proposals/` so the
> conversation history is committed alongside the code.
>
> _Last updated: 2026-05-03_

This document captures every unresolved scoping question for the next
two planned releases. Once each block is resolved, the answers move
into a release notes draft and the corresponding GitHub issue.

---

## Context: where we are

Two threads in flight from the v0.1.5 prep session:

1. **v0.1.6 — OSI semantic layer.** Add a portable, vendor-neutral
   semantic layer (Open Semantic Interchange spec, v1.0 finalized
   January 27, 2026 under Apache 2). Shipping a thin scaffolding layer
   on top of dbt marts, no native consumption yet.

2. **v0.1.7 — Cloud bucket checkpoint via DuckLake.** Add S3 / GCS /
   Azure Blob support so users can mirror local DuckDB state to a
   cloud bucket. Bucket-side format is **DuckLake** (DuckDB Labs
   lakehouse format, v1.0 released April 2026), giving cross-tool
   readability (Spark, Trino, Iceberg readers can read the underlying
   Parquet directly) and free snapshot history.

Both touch the same neighborhoods of code (`scaffolding/`, `register`
namespace, the wizard, `tycoon doctor`) so it's worth resolving the
cross-cutting questions before starting either.

## v0.1.6 — OSI semantic layer

### What's settled

- **Standard:** Open Semantic Interchange v1.0 — vendor-neutral YAML
  for datasets, metrics, dimensions, relationships, contexts. Founding
  partners include dbt Labs, Snowflake, Databricks, ThoughtSpot,
  BlackRock, Atlan, RelationalAI.
- **Source of truth:** dbt MetricFlow stays the source. OSI is a
  *published view* onto dbt. We don't reinvent metric definition; we
  emit a portable export.
- **Path A (lightweight scaffolding) is the v0.1.6 scope.** Path B
  (Nao consumes OSI) is deferred until dbt-core ships native OSI
  export. Path C (Rill metrics_views generated from the OSI source)
  is deferred until Rill joins the OSI consortium.

### Open questions

#### Command shape

1. **New top-level command or fold into `data analyze`?**
   - Option A: `tycoon semantics scaffold` — new namespace, parallels
     `data analyze` for dbt staging.
   - Option B: `tycoon data analyze --osi` — flag on the existing
     command.
   - Option C: Fully automatic — every `data analyze` run also emits
     OSI alongside dbt mart staging.

   _Lean: **Option A** for v1, with `data analyze` triggering it
   automatically in v2 if the UX feels right. New namespace makes
   the surface discoverable._

2. **Where does the OSI YAML live on disk?**
   - Option A: Per mart file — `dbt_project/models/marts/_osi.yaml`,
     `_osi.yaml` lives alongside the dbt mart it describes.
   - Option B: Per project file — `dbt_project/semantic/osi.yaml`,
     one file with all metrics.
   - Option C: Inside dbt project's `semantic_models/` directory (dbt
     convention for MetricFlow).

   _Lean: **Option B** — single project-level OSI YAML is what
   downstream consumers (Snowflake, ThoughtSpot, etc.) expect.
   Generating one-per-mart trades user simplicity for downstream
   complexity._

#### Generation heuristics

3. **What columns become OSI measures vs. dimensions?**
   - Numeric aggregates (`SUM`, `COUNT`, `AVG` columns) → measures
   - String / date / category columns → dimensions
   - Foreign keys (named `*_id`, `*_key`) → dimensions but flag as
     join points
   - Boolean → dimensions

   _Open: How aggressive should we be? E.g. is `widget_count`
   automatically a SUM measure, or do we just declare it and let
   the user add the aggregation? Auto-aggregation is convenient
   but error-prone (can't know the user's intent)._

4. **Sentinel + `--force` like the dbt staging scaffold?**
   - Generated OSI YAML starts with `# @generated by tycoon
     semantics`. Re-runs skip files where the user removed the
     sentinel.

   _Lean: yes, same pattern as `tycoon data analyze`. Free
   consistency._

#### Validation

5. **Should `tycoon doctor` validate generated OSI YAML against the
   v1.0 spec schema?**
   - The OSI consortium ships an official validator; we'd shell
     out to it (or vendor the schema and validate ourselves).
   - Doctor panel would show OK / WARN (schema-valid but partial) /
     FAIL (invalid).

   _Lean: yes, but as a `tycoon semantics doctor` subcommand
   rather than top-level doctor — the dependency is heavy and
   only matters for projects using OSI._

#### Auto-trigger

6. **Should OSI scaffold auto-trigger after `data transform run` /
   `data analyze`?**
   - Same UX pattern as the dbt staging auto-scaffold (sentinel
     guards re-run safety).
   - Default off in v1.6 (opt-in via config); default on in v1.7+
     once the heuristics have been validated by users.

   _Lean: ship behind `transform.auto_osi_scaffold: false` for
   v1, flip default later._

### Issue to file

> **Issue C — "v0.1.6: scaffold OSI semantic-layer YAML from dbt
> marts"**
>
> Acceptance criteria:
> - `tycoon semantics scaffold` writes a project-level OSI v1.0
>   YAML to `dbt_project/semantic/osi.yaml`
> - Sentinel-protected with `--force` override
> - Auto-detects mart columns → emits measures (numeric) and
>   dimensions (everything else)
> - `tycoon semantics doctor` validates against the OSI v1.0
>   schema
> - Documented in `docs/recipes/`
> - 5+ unit tests covering generation, sentinel, validation
>
> Linked: [Path B follow-up: Nao consumes OSI] (separate issue)

## v0.1.7 — Layer-aware data model ([#30][])

Filed as [#30][]. The bullets below summarize the open conversation; the issue body is canonical.

### What's settled

- **Scope: dbt + ingestion platforms only.** Tycoon classifies tables that come from one of two known creators — dbt models (via manifest) or ingestion tools (dlt, Fivetran, future Airbyte/Stitch). Tables created by humans, notebooks, ad-hoc SQL, or other orchestrators are not classified. Rationale: those are the two surfaces tycoon already governs, and they cover the medallion architecture. Hand-rolled tables falling through to "unclassified" is the right behaviour.
- **OSI scaffolding moves from prefix-matching to layer-awareness.** v0.1.6 finds marts via `mart_*` / `fct_*` / `dim_*` / `obt_*` prefix matching; v0.1.7 swaps this for `tycoon.layers.list_marts()`. No user-facing API change — the `tycoon semantics scaffold` CLI shape stays identical.
- **Two distinct sub-problems within one issue.** (1) source classification (Fivetran + dlt as a vendor-tagged class), (2) layer awareness (sources / staging / intermediate / marts as first-class). Source classification could ship without full layer awareness; layer awareness needs source classification as a prereq.

### Open questions

#### Surface ordering

1. **Does `tycoon data status` become layer-organized by default, or does `tycoon data layers` ship as a sibling command?**

   _Open. The status-by-source view is genuinely useful; the layers view is genuinely useful; I don't want to merge them just to have one command. Lean: ship `tycoon data layers` as a new sibling, leave `data status` source-focused. Promote layer-organized view to default in a future cycle if usage data warrants._

2. **Where does the layer override live in `tycoon.yml`?**

   For users whose dbt project doesn't follow the `models/staging/` / `models/intermediate/` / `models/marts/` convention. Two shapes:

   ```yaml
   # Option A: explicit folder mapping
   layers:
     staging: models/cleaned
     intermediate: models/joined
     marts: models/published

   # Option B: per-model overrides
   layers:
     overrides:
       fct_my_model: marts
   ```

   _Lean: **Option A.** Folder convention is the dbt community norm; per-model overrides are a footgun for projects with hundreds of models._

#### Vendor identification

3. **Is "vendor" tracked at runtime via lookups, or stored in the typed config?**

   - Runtime lookup: at `tycoon data status` time, ask `tycoon.yml` (for dlt source names) and the Fivetran API (for connector schema names) to build the vendor mapping. Costs an API hit on every render.
   - Typed config: a `sources:` block in `tycoon.yml` that explicitly lists every source schema and its vendor. Stale-state risk if a Fivetran connector is renamed without updating the file.

   _Lean: **runtime lookup with a 60-second cache** in `.tycoon/sources_cache.json`. Vendor info changes rarely; a cache deals with the API-hit concern without introducing config-staleness footguns._

#### dbt-less projects

4. **What does layer awareness mean when `transformation: none`?**

   _Lean: **source layer only**, gracefully skip the rest. `tycoon data layers` shows just the sources panel. `tycoon doctor` doesn't run the "every source has a staging model" check. Documented as expected behaviour, not a degraded mode._

#### Backup integration (v0.1.7 cross-cutting)

5. **Does layer classification affect what gets backed up?**

   The backup story below distinguishes "raw" from "warehouse" — but the layer model is more granular. Should `backup.skip_raw: true` become `backup.layers: [marts, intermediate]` in v1.7?

   _Lean: **yes, but as a v0.1.8 follow-up**. v0.1.7 ships `skip_raw` from the original spec; layer-granular backup is a refinement once layer classification has shaken out._

### Acceptance criteria

See [#30][] — preliminary criteria there, will be refined as the design questions resolve.

## v0.1.7 — Cloud bucket checkpoint via DuckLake

### What's settled

| Question | Answer |
|---|---|
| **Q1 — Backup semantics** | (a) Backup mode — single canonical state on the bucket. DuckLake gives us free history as a side effect; we treat it as backup at the UX layer. |
| **Q2 — Layers backed up** | Warehouse + raw + metadata DBs. Default includes raw; opt out via `backup.skip_raw: true`. |
| **Q3 — Trigger** | Hybrid: auto-sync after pipeline ops (default-on); manual `tycoon data backup` available; opt-out via `backup.auto_sync: false`. Smart-skip via mtime so unchanged files don't re-upload. |
| **Q4 — Restore UX** | Smart-default: if local files don't exist, download to canonical paths; if they do, download to siblings + require `tycoon data backup promote` to swap. `--force` and `--safe` flags for explicit overrides. |
| **Q5 — Provider abstraction** | DuckLake's httpfs handles S3 / GCS / Azure / R2 via uniform URI schemes. Auth via env vars. |
| **Bucket-side format** | DuckLake (always — user's explicit call). Local stays standard `.duckdb` files. |
| **Catalog placement** | Inline in the bucket prefix at `_lake_catalog.duckdb`. Bucket is self-contained. |

### What the architecture looks like

```
LOCAL                              BUCKET (DuckLake on S3)
data/files_raw.duckdb         ⇄    s3://my-bucket/myproj/
data/warehouse.duckdb              ├── _lake_catalog.duckdb
.tycoon/metadata.duckdb            └── data/
                                       ├── raw/        (Parquet files…)
                                       ├── warehouse/  (Parquet files…)
                                       └── metadata/   (Parquet files…)
```

```yaml
# tycoon.yml
backup:
  destination: s3://my-bucket/myproj/
  auto_sync: true
  skip_raw: false
  retain_snapshots: all   # all | <N>_days | latest
```

CLI:
- `tycoon register backup --to s3://my-bucket/myproj` — initialize
- `tycoon data backup` — manual push
- `tycoon data backup pull` — restore (smart default)
- `tycoon data backup promote` — swap restored siblings → canonical
- `tycoon data backup status` — local/lake state diff
- `tycoon data query "..." --attach-lake s3://...` — direct read

### Open questions

#### Catalog & layout

1. **Catalog DB choice for v1: DuckDB-on-S3 (single-writer, simple)
   or PostgreSQL (multi-writer, requires server)?**

   _Lean: **DuckDB-on-S3 inline.** Single-writer is correct for
   tycoon's current single-user backup use case. Postgres deferred
   to v2 if multi-user team setups become a real ask._

2. **Should `tycoon register backup` support catalog living
   separately from data?** (e.g. team setup: data on S3, catalog in
   shared Postgres)

   _Lean: **no for v1**, single-config "everything in the bucket."
   Add as a v2 opt-in when the team-multi-user use case materializes._

3. **Layout — three schemas inside one DuckLake (proposed) vs. three
   separate DuckLakes (one per layer)?**

   _Lean: **three schemas in one DuckLake.** Atomic multi-table
   commits; single connection to manage; user sees one bucket
   prefix. The alternative buys nothing._

#### Snapshot retention

4. **What's the default retention?**

   - "Keep all" → bucket grows monotonically; storage cost proportional
     to total data churn.
   - "Latest N" → caps history but loses time-travel beyond N.
   - "Older than X days" → automatic compaction; loses time-travel
     beyond X.

   _Lean: **keep all** by default — DuckLake snapshots are tiny
   (only the parquet data scales with churn). Add
   `backup.retain_snapshots: <N>_days` as opt-in for users with
   high-churn warehouses._

5. **Should `tycoon data backup` expose time-travel even though Q1 =
   backup semantics?**

   - Q1 said "no time-travel" but DuckLake gives it for free.
   - We could expose it via `tycoon data backup pull --as-of
     2026-04-15` — costs almost nothing to plumb.
   - Risk: users start to rely on it, which contradicts "single
     canonical state" mental model.

   _Lean: **don't expose in v1.7**, evaluate user demand for v1.8._

#### Encryption / security

6. **Encryption-at-rest for v1?**
   - SSE-S3 (default for S3): nothing tycoon needs to do.
   - SSE-KMS / customer-managed keys: requires per-bucket config.
   - CSE (client-side): tycoon encrypts before upload; needs key
     management.

   _Lean: **rely on bucket-side defaults for v1**. Document SSE-KMS
   as user responsibility (set on bucket, not in tycoon). Defer
   client-side encryption indefinitely._

7. **Credential handling — env-var only, or also support AWS profile
   / GCS service-account file paths?**

   _Lean: **env vars first** (matches what `tycoon data sync` does
   today). Add profile support if users complain._

#### Multi-bucket / multi-target

8. **Should one project be able to back up to multiple buckets?**
   (e.g. dev → personal S3, prod → org S3)

   _Lean: **no for v1**, single destination. Add `--to <override>`
   on the CLI for one-off operator overrides without persisting.
   Multi-target as a v2 design._

#### Bandwidth / scale

9. **Big-warehouse scale: chunked uploads, parallelism, resume?**
   - DuckLake's INSERT-into-DuckLake is single-stream by default.
   - For warehouses > 10 GB, sync time can be minutes.
   - DuckDB has `SET threads = N` for parallelism but bucket I/O
     is the bottleneck.

   _Lean: **single-threaded for v1**, document as a known limit.
   Revisit if users surface complaints._

10. **What happens when the network drops mid-sync?**
    - DuckLake's transaction semantics should make this safe — the
      catalog won't commit until the data write succeeds.
    - But partial Parquet uploads might leave orphan files.

    _Open: needs a probe. Document the recovery story (`PRAGMA
    ducklake_repair`?) before shipping._

#### Lifecycle / cleanup

11. **What happens when the user renames the project?**
    - The bucket prefix is configured in tycoon.yml; renaming
      doesn't move the data automatically.
    - `tycoon register backup --to <new-prefix>` would re-init at
      the new location, leaving old data orphaned.

    _Lean: **document the manual move** (`aws s3 mv`); add
    `tycoon data backup migrate <new-prefix>` as a v1.8 nicety._

12. **Bucket-side lifecycle policies (auto-expire old snapshots)?**
    - Easier to manage bucket-side via S3 Lifecycle Rules.
    - Tycoon could emit a sample policy in
      `docs/recipes/cloud-backup-lifecycle.md` but not own the
      configuration.

    _Lean: **document, don't manage.** Surface a sample policy
    YAML for each provider in the recipe._

#### Doctor integration

13. **`tycoon doctor` panel for backup state?**
    - Surfaces: in-sync / out-of-sync / never-synced / sync-failed.
    - For out-of-sync, shows local mtime vs. last successful sync.
    - For sync-failed, shows the error from the last attempt.

    _Lean: **yes, ship in v1.7.** Critical for "did my last sync
    succeed?" without rummaging through logs._

#### v0.1.3 lock blocker — does it apply?

14. **Does the v0.1.3 DuckLake catalog lock issue affect this
    workflow?**

    Memory says: SQLite-backed DuckLake catalogs took an OS-level
    exclusive lock when any process held them attached, even
    `READ_ONLY`. Two readers OK; reader + writer not.

    For v1.7 backup workflow:
    - **Single writer (tycoon's sync), brief windows.** Conflict
      surface is small.
    - **No long-running readers** holding the catalog open.
    - **Local `.duckdb` files are unchanged** — Rill keeps reading
      those, never touches the S3 DuckLake.

    _Conclusion: **the lock blocker doesn't apply to v1.7 backup**.
    It still applies to the deferred "Rill DuckLake live connector"
    feature. Worth retesting against DuckLake v1.0 + DuckDB-backed
    catalog (instead of the SQLite catalog originally probed in
    v0.1.3) to see if the lock issue persists at all._

## Cross-cutting questions

### CC-1. Should the OSI YAML live in the bucket alongside the data?

If a teammate restores the v0.1.7 backup on a fresh machine, do they
get the metric definitions (OSI YAML) or just the raw mart data?

- **Yes, include OSI in backup:** simpler "everything you need in one
  place" model. Costs nothing — OSI YAML is small.
- **No, OSI lives in git:** stricter separation of "code" (git) vs.
  "data" (bucket). Teammate clones the repo to get OSI.

_Lean: **no — OSI in git only**, matching the broader "code in git,
data in bucket" boundary. Document the assumption. The teammate
always clones the repo first; the bucket is for data state, not
project state._

### CC-2. Implementation sequencing

Both features are scoped to be roughly the same size (~1-2 weeks of
focused work each). Three sequencing options:

- **A: OSI first (v0.1.6) → backup (v0.1.7)**

  Pro: OSI is structurally simpler (no I/O, no auth, no error modes
  related to network). Lower-risk first.

  Pro: OSI gives users immediate value at the moment dbt-core ships
  native OSI export (then Path B Nao consumption is a v0.1.8 add-on
  — already aligned).

  Con: Backup is the higher-impact feature (most users want
  cloud-mirroring more than they want a portable semantic export).

- **B: Backup first (v0.1.6) → OSI (v0.1.7)**

  Pro: Higher user impact first. Cloud backup is what users will
  actually adopt.

  Pro: OSI then naturally extends to "the OSI YAML is in the bucket
  too" if we revisit CC-1.

  Con: Backup is more complex (network, auth, lake catalog,
  per-provider quirks); higher chance of v0.1.6 slipping.

- **C: Interleave**

  Land OSI scaffolding first as a lightweight piece, ship behind a
  feature flag, then ship backup with OSI integration as a v0.1.6
  combined release.

  Con: violates the "small focused releases" pattern from v0.1.x;
  high coordination overhead.

_Lean: **Option A (OSI first).** Smaller risk, shipped sooner,
unblocks the dbt-Labs OSI export discovery for users who want to be
on the bleeding edge. Backup follows in v0.1.7. We get two clean
releases instead of one ambitious one._

### CC-3. `tycoon register` namespace consistency

After v0.1.7, the register namespace would have:

- `tycoon register dbt`
- `tycoon register warehouse`
- `tycoon register rill`
- `tycoon register llm` (added in v0.1.5)
- `tycoon register backup` (proposed for v0.1.7)
- `tycoon register semantics` (alternative for OSI?)

Each "registers" an external resource and triggers a
post-registration setup. Consistent and discoverable.

_Open: should OSI be `tycoon register semantics` instead of
`tycoon semantics scaffold`? It's not really an external
resource — the file lives in the project — so the analogy is
weaker. But the namespace consistency would be appealing._

_Lean: **keep OSI under `tycoon semantics` namespace** (not
register), since the YAML output is project-local. Register is for
"point tycoon at a thing that exists outside." The proposal is
fine as-stated._

## Glossary / context

- **OSI** — Open Semantic Interchange. Current spec **v0.1.1** dated
  2025-12-11 (this proposal originally claimed v1.0 finalized
  2026-01-27 — that was wrong, corrected here for the record).
  Vendor-neutral YAML for semantic layer constructs (datasets,
  metrics, dimensions, etc.). Apache 2 licensed. https://open-semantic-interchange.org/
- **DuckLake** — DuckDB Labs' lakehouse format. v1.0 released
  2026-04. Combines Parquet for data + a SQL-database catalog for
  metadata; supports concurrent multi-writer via the catalog DB's
  transaction semantics. https://ducklake.select/
- **The v0.1.3 lock blocker** — SQLite-backed DuckLake catalog
  rejected concurrent reader + writer access on the same catalog
  file. Documented in user memory `reference_rill_yaml_format.md`.
  Affects the deferred "Rill DuckLake live connector" feature; does
  NOT affect the v0.1.7 backup workflow proposed here.
